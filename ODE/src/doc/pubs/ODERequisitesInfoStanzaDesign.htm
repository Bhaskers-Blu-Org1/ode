( P | pre )
pre-requisite -
AIX A prereq indicates both a functional and an installation order dependency of one fileset on another fileset. Pre-requisites should be used as little as possible in packaging. 
HP A pre-requisite dependency requires another software object to be installed (configured) in order for the first fileset to be installed.
MVS In SMP/E, a SYSMOD that must be installed before or along with another SYSMOD in order for that other SYSMOD to be successfully installed. It is defined by the PRE operand on the ++VER statement.
RPM (Prereq)
Solaris A package that depends on the existence of another package.
( C | co )
co-requisite -
AIX A coreq indicates only a one-way functional dependency of one fileset on another fileset. Co-requisites are the most desirable requisite to use in the case of a functional dependency since a coreq only indicates that both filesets must be installed on the system; no order to the install is implied. A common misunderstanding with coreqs is that they are not two-way as the name may imply.
HP
1.) A corequisite dependency states that a fileset requires another software to operate correctly.
2.) A corequisite dependency requires another software object to be installed in order for the dependent fileset to be usable.
MVS corequisite SYSMODs. SYSMODs each of which can be installed properly only if the other is present. Corequisites are defined by the REQ operand on the ++VER statement.
RPM. (Requires) The requires tag is used to warn RPM that the package needs to have certain capabilities to operate properly.  These capabilities refer to the name of a package or to a virtual package provided by one or more packages that use the provides tag.  A package that requires certain ``capabilities''.  Manual dependency requirements are always represented in terms of packages..
Solaris N/A.
( L | ins )
install requisite -
AIX An instreq is used in cases where one fileset should be installed only IF another fileset is installed.  It doesn't make sense to install the documentation for a package unless the "core functionality" is also installed.  But the documentation will not prereq the "core functionality" it will instreq it.
HP N/A.
MVS N/A.
RPM N/A
Solaris N/A.

reverse requisite -
AIX N/A.
HP N/A
MVS N/A
RPM N/A.
Solaris A condition when another package depends on the existence of your package.  Use the reverse dependency type only when a package that cannot deliver a depend file relies on your package.
( I | if )
if-requisite -
AIX An ifreq is probably the most complicated form of requisite and is used primarily with updates. An ifreq is best thought of as a conditional coreq. The ifreq is similar in behavior to a coreq, in that, if one update ifreqs another update then both updates must be installed at the same time or the target update must already be installed, with the order of installation not being important. The conditional part is that the second fileset must only be installed at the specified level if it is already installed on the system at some lower base or specified level. Otherwise, the ifreq is ignored until the second fileset is installed on the system, at which time lppchk can be used by the customer to tell them if there are any missing updates. The ifreq does NOT allow installp to add the required fileset to the list of filesets to install (or update) even if the -g flag is used.
HP N/A.
MVS (IFREQ) A conditional requisite. Conditional requisites are specified on the REQ operand of the ++IF statement.
RPM N/A
Solaris N/A
( X | npr )
negative pre-requisite - 
AIX N/A.
HP (exrequisites) (Not yet implemented.) A list of dependencies on software that may not be installed when this software is installed. See also the ancestor, corequisites, and prerequisites attributes..
MVS In SMP/E, a function that is mutually exclusive with another function. It is defined by the NPRE operand on the ++VER statement.
RPM N/A.
Solaris (incompatible)
1.) Implies that the existence of the indicated package or version is incompatible.
2.) (Incompatible package) A package that is incompatible with the named package.
( S | su )
supercedes -
AIX The fileset.supersede file is optional and used for both install images and updates.  When used with an install image, the fileset.supersede should list the one fileset that this image completely replaces.  For updates, the fileset.supersede file contains a list of the fileset and levels (with fix field smaller than this one) that are superseded by this fileset update.
HP (Ancestor)An "attribute" which signifies the name of a previous version fileset used in the "Match-What-Target-Has" functionality. If the match_target option is set to true, the system will match the ancestor fileset name to the new fileset name.
RPM N/A. 
MVS superseding SYSMOD. In SMP/E, a SYSMOD that contains all the functions in another SYSMOD and is recognized as the equivalent of that other SYSMOD. The superseding SYSMOD uses SUP operand on its ++VER statement to specify the superseded SYSMOD.
Solaris (compver) A method of specifying package backward-compatibility.
sup

AIX N/A.
HP Patches that supersede all previous patches to a given fileset.  Used when a patch is replaced by (or merged into) a later patch. The attribute indicates which previous patches are replaced by the patch being installed or copied. This attribute value is a list of software specifications of other patches that this patch supersedes.
MVS N/A.

group requisite - Not Implemented by RequisitesInfo stanza in ODE.
AIX This is simply a format which can be used to indicate to the installp command that one or more of the following requisites are required to continue. An example of this would be where an LPP requires either the Netscape Navigator or Communicator product on the system as a co-requisite. This would be shown as :

rw
requisite webs - Not Implemented by RequisitesInfo stanza in ODE.
AIX In packaging it is sometimes necessary for an LPP to split core function into small filesets for ease of update, but the overall function of the LPP will not work without all the function installed on the system. This is a case where the LPP should split the core function into these smaller filesets, and then use requisite webs to ensure that all the core function is installed at the same time if any one piece is requested. Be sure to not carry these requisite webs over from install images to update images. Also take care to ensure that the web does not become circular.

Install Requisite (instreq)
An instreq is used in cases where one fileset should be installed only IF another fileset is installed. For example, the messages for some function should only be installed if that function is already installed, or in the process of being installed. The messages are of no use to the customer and only waste space on the customer system if the function which uses them is not installed.
supercedes and superceding patches
An obvious typo has been carried over from MVS documentation where supercedes has been spelled supersedes.  Also an incompatibility exists where supercedes on HP is for patches only and on other platforms it is for install images and updates.  Probably the best thing to do would be to use both and force other pkgtools to coply with HP's new tag.  The generators will enforce the correct syntax and keyword for each platform.
control scripts
HP. Control Scripts  Optional, vendor- or administrator-supplied scripts that are run during swinstall and swremove. Includes checkinstall, preinstall, postinstall and configure scripts for swinstall; the checkremove, unconfigure, preremove, and postremove scripts for swremove; and the verify script for swverify
Solaris. You are not required to write any installation scripts for a package. The pkgadd command performs all the actions necessary to install your package, using the information you supply with the package information files. Any installation script that you provide is used to perform customized actions beyond those executed by pkgadd.

# Files:

  • Checkinstall scripts are executed during the Analysis phase of a swinstall session. The pathname of the script being executed is:
    [$ {SW_CONTROL_DIRECTORY}checkinstall]
  • A checkinstall script must not modify the system.
  • A checkinstall script determines whether the product/fileset can be installed by performing checks beyond those performed by swinstall. Example checks include checking to see if the product/fileset is actively in use, or checking that the system run-level is appropriate.
  • If you are using a request script as part of the install, the checkinstall script should:
      Verify that the response file exists.
      Prevent swinstall from "hanging" if:
        A script tries to read a response file that does not exist.
        The install or configuration relies on information in the missing response file.

The checkinstall script is intended to perform basic checks on a file system that would not be normal for the pkgadd command. For example, it can be used to check ahead to determine if any files from the current package are going to overwrite existing files, or manage general software dependencies (the depend file only manages package-level dependencies).

Here is a sample checkinstall script:

# checkinstall script for SUNWcadap

#

# This confirms the existence of the required specU database

# First find which database package has been installed.

pkginfo -q SUNWspcdA # try the older one

 

if [ $? -ne 0 ]; then

   pkginfo -q SUNWspcdB # now the latest

 

   if [ $? -ne 0 ]; then # oops

      echo "No database package can be found. Please install the"

      echo "SpecU database package and try this installation again."

      exit 3 # Suspend

   else

      DBBASE="`pkgparam SUNWsbcdB BASEDIR`/db" # new DB software

   fi

else

   DBBASE="`pkgparam SUNWspcdA BASEDIR`/db" # old DB software

fi

 

# Now look for the database file we will need for this installation

if [ $DBBASE/specUlatte ]; then

   exit 0 # all OK

else

   echo "No database file can be found. Please create the database"

   echo "using your installed specU software and try this" echo "installation again."

   exit 3 # Suspend

fi
