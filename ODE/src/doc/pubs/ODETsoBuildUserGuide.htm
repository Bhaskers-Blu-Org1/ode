About this Document

This document is intended to give the reader the necessary information to build ASM, COBOL, PLS and PLX parts on the Time-Sharing Option (TSO) side of a z/OS System using Open Development Environment (ODE).  TSO is the shell, or user interface, by which users interact with the operating system.

Note:  The current ODE TSO build process only works in a z/OS USS ODE environment.

See the ODE Documentation Index for a complete list of ODE documents and their purposes. It is recommended that the ODE Build Concepts and ODE Build Reference documents be read before running the TSO build process..

Once a build is done, one or more ODE passes can be used to create a package.  More information about packaging can be found in the ODE Packaging Reference, as well as the ODE Packaging Tutorial.

Introduction

ODE primarily has been the build tool of choice for z/OS USS based products. It supports building and packaging of C/C++ and Java code in the z/OS USS environment. It is conceivable that some USS based products may contain code that need to be built in the TSO environment. ODE currently supports the building of parts on the TSO side of a z/OS system for ASM, COBOL, PLS and PLX.

Getting Started

This section describes what an ODE user will need do inorder to perform TSO builds.  To start using the TSO build feature of ODE, there are several things that need to be done like verifying access to the remote machine, modifying the makefiles, establishing any user-defined rules for the build_all targets and creating a netrc dataset on the remote machine.

Note:  The TSO build rules are not turned on during the normal ODE build process, unless specified by the user. Therefore, the variable USE_TSO_RULES is not on by default.  If the user wants to perform TSO builds on the z/OS system, then the variable must be set (.i.e. USE_TSO_RULES=1).  If  the variable is undefined, empty (i.e. USE_TSO_RULES= ) or set to zero  (i.e. USE_TSO_RULES=0),  the TSO build rules will not be utilized and TSO builds can't be performed.

Note:  To run the TSO build process with the BBEXAMPLE backing build provided by ODE, please add the following line to Buildconf in the bbexample/src directory:

    replace setenv SOURCEBASE ${source_base}
 

Valid TSO Userid and Password

The user must have a valid TSO userid and password on the remote machine inorder to request the use of its resources.  This information is also used to submit Job Control Language (JCL) jobs and to copy data via FTP between the two machines.  If you don't have a TSO userid  and/or password at this time, have your TSO administrator assign one to you.
 

Set Classpath

An additional step must be taken to set up the environment for building TSO parts.  The ODE jar file (i.e. ode5.0_b4_tools.jar) must be added to the CLASSPATH variable, for instance on Unix systems, enter the following:

  export CLASSPATH=/ode/classes/ode5.0_bnum_tools.jar:$CLASSPATH

where /ode/classes is the location of the tools jar file on your system, and num is the build number.
 

Prerequisite for Dependency Generation

The dependency information will come from the information provided by the compiler.  This information is interpreted and used to produce the correct ODE format (.u file) on the HFS filesystem.  Depending on the language (i.e. ASM), the user will have to identify the appropriate Monitor Exit (i.e. SDYNASM Exit) for dependency generation.  The user will also need to define the variable DEPENDENCIES in their makefile, if they want to have this capability turned on.

Note:  ODE does not currently produce any dependencies for COBOL due to the lack of dependency information generated by the compiler (no Monitor Exit capability).  The following message will appear during file dependency phase:

****************************************************
* Dependencies for Cobol - not currently supported *
****************************************************

Therefore, all COBOL files in the traversed directory will be recompiled.
 

JobCard

A local job card file (see the example below) used to provide job submission information to MVS, must be present on the system that the build is to run on.  Consult your MVS documentation for more information on the creation and use of job cards.  The sub-parameters are positional, to omit one sub-parameter, the positional comma after that sub-parameter must be coded.

//WMATHIS1 JOB (349077,,,,10,N),
//  'WMATHIS',TIME=1440,NOTIFY=WMATHIS,
//  MSGCLASS=O,
//  MSGLEVEL=(1,1)
//*MAIN SYSTEM=MVSC,ORG=ANYLOCAL,
//*CLASS=VPRLS,
//*USER=WMATHIS,HOLD=NO
//*IBM INTERNAL USE ONLY

Note:  If the computer operating system determines that either the time or lines estimates will be exceeded, it will abend the job with a system completion code.
 

NETRC

On the remote machine, create a dataset called NETRC under your userid.  If the variable BLD_NETRC_FILE is set (i.e. BLD_NETRC_FILE=WMATHIS.FTP(NETRC) ), then it will be used by FTP for automatic login on the remote host. For each remote host, the file contains an entry that describes the login data for the user on that host.  The NETRC dataset is used to hide the userid and password during the FTP process.  However, some users may want to bypass the NETRC file setup.  Therefore by default, ODE does not require the user to have this variable BLD_NETRC_FILE set.  If this variable is not set, then the user's id and password from the makefile variables will be utilized for the FTP process.

Note:  Bypassing the NETRC setup is not recommended by ODE for security reasons.  Without the NETRC setup, the user's id and password will be visible in the JCL located in the obj directory.

The first part of the NETRC is filled up with host-definitions. Although there are others, the following tokens (machine, login and password) are all that is needed for the dataset and they may be separated by spaces, tabs, or new-lines.

This is an example of a NETRC dataset:

WMATHIS.FTP(NETRC) contains:

machine raljes3c.raleigh.ibm.com login wmathis password xxxxxxxx
machine raljes3c login wmathis password xxxxxxxx

What it is saying now is nothing more than "If you connect to raljes3c or raljes3c.raleigh.ibm.com, login as wmathis and use xxxxxxxx as password." Note:  If a user specify BLD_REMOTE_HOST = raljes3c in the makefile, that short hostname should appear in the NETRC file also.  If omitted, the user may encounter a login and/or password error that causes the FTPFILE step to fail.

User-Defined Rules

This section describes how to override and/or enhance rules established in the ODE rules files. The files in src\rules_mk establish all of the rules that ODE incorporates. However, if it is intended to override one of the rules or variable settings specified in these files, it can be accomplished by creating user-defined rules files myproj.rules.mk and (sometimes) myproj.passes.mk.

   1.In the src/Makeconf file, a variable PROJECT_NAME is defined to be myproj. This variable can be specified by editing Makeconf and modifying the value of PROJECT_NAME. Assume for this example that
          PROJECT_NAME = myproj
   2.In the src/rules_mk directory, create a file myproj.rules.mk.
   3.Add any rules or variable definitions to myproj.rules.mk. This file will be processed last by ODE, allowing rules to be redefined and variables to be overridden or appended to, without changing any of the ODE-delivered rules files.
   4.If new passes or actions are created by the user, some rules also have to be put in the myproj.passes.mk file.  This file is processed just after ODE defines the predefined actions and passes.

See the Adding Another Pass section for an example of modifying the myproj.rules.mk and myproj.passes.mk files.  The ODE Common Makefiles document also has information about adding passes.
 

JCL Information

This section describes the utilization of the JCL Templates by ODE.  ODE generates dynamic JCLs from JCL templates shipped with ODE. These dynamic JCLs are submitted via FTP to the user's remote machine and the jobs perform the following task:  dataset allocation, file copying, compilation and dataset deletion.  To facilitate the build on the TSO side of an MVS system, the user can permit the default option (BLD_USING_ALLOC_REXXEXEX=NO) to create multiple Partitioned Data Sets using IEFBR14 or allow the ODE REXX script to drive the allocation functions by setting the variable BLD_USING_ALLOC_REXXEXEX=YES.

Note:  The ODE REXX script must be installed per the instructions detailed in the Installation and Adminstration Guide.

A set of temporary libraries will be created and deleted during the TSO Build process.  The user's TSO userid will be used as the HLQ of the libraries, if the variable BLD_HLQ is not set.  The following libraries will be created:

If  BLD_HLQ is set , then                                If  BLD_HLQ is null or set to the TSO userid , then

  HLQ.USERID.SBOXNAME.SRC             USERID.SBOXNAME.SRC
  HLQ.USERID.SBOXNAME.MACLOCAL        USERID.SBOXNAME.MACLOCAL
  HLQ.USERID.SBOXNAME.MACEXPT         USERID.SBOXNAME.MACEXPT
  HLQ.USERID.SBOXNAME.O               USERID.SBOXNAME.O
  HLQ.USERID.SBOXNAME.LISTING         USERID.SBOXNAME.LISTING
  HLQ.USERID.SBOXNAME.CLRPRINT        USERID.SBOXNAME.CLRPRINT
  HLQ.USERID.SBOXNAME.UNIDATA         USERID.SBOXNAME.UNIDATA

Partitioned Data Sets:
  SRC         - contains the source code
  MACLOCAL    - contains the local macros
  MACEXPT     - contains the export macros
  O           - contains the compile objects
  LISTING     - contains the compile listings
  CLRPRINT    - contains the dependencies for the PLS and PLX objects
  UNIDATA     - contains the dependencies for the ASM objects

NON-ODE JCL Templates

A user may request to utilize their own JCL Templates (ASM, COBOL, PLS or PLX) during the TSO build process.  ODE reads the templates, incorporates the additional JCL information, places the data from the source response file (mvssrclist.rsp) into the build control file (mvsbldlist.rsp) and utilizes the appropriate default or user specified syslib makefile variables information to create the dynamic JCLs.

Note:  To utilize your JCL templates,  the templates need to be rename to one of the following:

  • usrasm.jcl       (This is the new User ASM Template name)
  • usrcobol.jcl     (This is the new User COBOL Template name)
  • usrpls.jcl         (This is the  new User PLS Template name)
  • usrplx.jcl         (This is the new User PLX Template name)
and placed in the rules_mk directory.

ODE TSO Build Process

This section describes the ODE TSO Build Process.  In performing a build, files to be compiled are in subdirectories in the src area of a sandbox. The build places the build control file, the object, listing and jcl files in the obj area of the sandbox.

A typical build process involves creating a sandbox, populating the sandbox with the files which need to be compiled and then running the mk or build command.   This pass builds the modified source files.

The mk or build command builds the default target, which is build_all. The build_all target then triggers the targets export_all and comp_all, which first does the export and then the compilation.

Below is an example of a makefile for building PLX parts.

Dependency Generation

This section describes the dependency generation for the TSO Build process. Dependencies are automatically created when mk is called, if the variable DEPENDENCIES is defined and the Monitor Exit is utilized for the appropriate language. In the Makeconf file shown previously in this document, DEPENDENCIES is defined. The Monitor Exit is used to generated the dependency file for each compiled source part.  Each temp (i.e. test.tmp and test2.tmp) file is copied to HFS and used to create the ODE dependency file (depend.mk).  Note:  The temp (i.e. test.tmp and test2.tmp) files are deleted after the depend.mk file is generated.

To verify that the dependencies were generated properly, examine the contents of the depend.mk file.

Contents of depend.mk in the /u/wmathis/sted/test/d13839/obj/mvs390_oe_2/plxExample directory:

#
# dependents of test2.o:
test2.o:  ${MAKETOP}plxExample/test2.plx
test2.o:  ${MAKETOP}../export/mvs390_oe_2/usr/incls390/d.mac
#
# dependents of test.o:
test.o:  ${MAKETOP}plxExample/test.plx
test.o:  ${MAKETOP}/plxExample/a.mac

This will cause ODE to compare all of the appropriate files' timestamps to determine which source files need to be recompiled.  The mk command will check the files in the sandbox(es) and the backing build automatically.

Note:  Currently, ODE is not able to generate a dependency file for COBOL because of the lack of a Monitor Exit to generate the dependencies.

When a developer needs to add or delete source files to/from the project, the makefile.ode file in those directories will have to be changed to add or remove the filenames appropriate variables.
