Highlighting Conventions

The style conventions used in ODE command syntax are described in How to Read Syntax Statements. The following highlighting conventions are also used in ODE documentation:
Bold
Actions, backing build areas, commands, directories, drive names, execs, flags, filenames, field names, machine names, names of sandboxes, parameters, passes, targets, appear in bold. Valid abbreviations for commands are also in bold.
Italic
Arguments, options and variables whose names or values must be supplied by you appear in italics. Italics are also used for emphasis and for titles of books.
Monotype
Examples of specific data values, examples of expected output, messages, or information that you should type appear in monotype.

Installing and configuring ODE

IBM ODE provides these components for installation:
  • Executable commands for both developers and builders, available as zip files or install images
  • Base set of common Makefile rules, used in both the backing build example and many ODE projects
  • A backing build example primarily for builders to get started
  • A generic set of configuration files (Makeconf, Buildconf, etc.) to start with when creating a new ODE project.

Obtaining ODE

All of the various ODE components can be obtained using the following steps:
  1. Go to the ODE home page: http://w3.ode.raleigh.ibm.com/
  2. Click on the Download link on the left margin.
  3. Click on ODE 5.0.
  4. Click (or Shift-click, if you have trouble downloading .jar files) on the desired files to download.  For new users, this will normally include the archive(s) of the tools for the desired platform(s) to use ODE on (e.g., %release_name%_bnum_rios_aix_4.zip, where num is the desired build number), the rules and bbexample archives, and the tools.jar archive.

Installing and configuring the ODE commands using zip files

The following information applies to the installation of the ODE commands in all the supported environments. These instructions assume that you will install ODE in the /ode directory. However, this is not necessary. ODE can be installed in another directory.
Note:  An unzip utility needs to be used for the installation process.  The instructions will assume the use of the JDK jar command to unzip the zip files.

AIX

To extract the file %release_name%_bnum_rios_aix_4.zip (where num is the build number), enter the following lines at the command prompt: 
  %mkdir -p /ode/aix
  %cd /ode/aix   
  (copy %release_name%_bnum_rios_aix_4.zip to /ode/aix)
  %jar -xvf %release_name%_bnum_rios_aix_4.zip
To assign permissions to read-write-execute for owner, read-execute for group, and read-execute for others enter the command: 
  %chmod 755 /ode/aix/*
The PATH and LIBPATH environment variables should be set as the following: 
  %export PATH=$PATH:/ode/aix
  %export LIBPATH=$LIBPATH:/ode/aix
To use the packaging tools, set the CLASSPATH environment variable to: 
  %export CLASSPATH=$CLASSPATH:/ode/classes/%release_name%_bnum_tools.jar
or rename the jar file to something more common.  By renaming the file, it will allow to easily upgrade your version of ODE without resetting your CLASSPATH.  For example, you could rename %release_name%_bnum_tools.jar to odetools.jar.

Upgrading the ODE commands

ODE commands can be upgraded by organizing an install tree so you can easily switch between different releases and builds of ODE commands.

Upgrading ODE commands involves removing the old installation and replacing it with either a new installation or creating a new subdirectory for a new release/build.

For example, you can have the installation directory /ode/release_name/build_num/machine_name for release 5.0 build 20031202.0300 on AIX. You would install to /ode/5.0/20031202.0300/rios_aix_4. Then, you would have links to /ode/latest-release/latest-build/rios_aix_4. 

Appendix A. Restrictions

Restrictions in UNIX environments

Directories in sandboxes and backing builds must be real directories; they can't be symbolic links.  When the directories are symbolic links, it causes the recursive mk tool to lose track of where it was and where it is now.  Since the rules heavily depend on the mk tool to properly set relative path information, symbolic links can cause unwanted results.  Since UNIX-type platforms support symbolic links, this restriction is limited to UNIX.  Use the mklinks tool to create symbolic links to files and create the directories that contain the links.

OS/400 environment (Qshell)

  • Two code sets are supported for all files which ODE tools reads and writes: 037 (EBCDIC) and 437 (IBMPC ASCII).  The code set which Java's jar command uses when extracting files, 819 (ISO ASCII), is not supported.  When ODE creates configuration files for the first time (e.g., .sandboxrc), they will always be in code page 037; however, when updating existing files, their code pages will remain as whatever they are currently set to.
  • So that ODE may work in both case-sensitive (/QOpenSys) and case-insensitive (/) file systems, ODE behaves in a mostly case-sensitive manner.  Although the file accesses that are handled by the system API (e.g., the stat() function) will dictate case sensitivity for many operations, all ODE-internal pathname and string comparisons, pathname sorting, and related activity will behave in a case-sensitive manner.
  • The equivalent of Ctrl-C, which in Qshell is SYSREQ-2, will not kill child processes.  This effectively means that mk cannot be interrupted in the same way as on other operating systems.  The F3 (Exit) key can of course be used to terminate the Qshell environment altogether, which will kill all running jobs.
  • mk cannot directly read/write the "members" of an archive library (touch members, use members as targets/dependents, etc.).

Restrictions and differences in non-UNIX environments

The following summarizes restrictions for non-UNIX platforms such as OS/2, Windows, etc.

OS/2 environment

  • All configuration files must use the UNIX-style directory slashes (/) rather than the DOS-style slashes (\). For example, the OS/2 directory c:\tmp must appear in an ODE resource configuration file as c:/tmp.
  • Drive letters are legal in absolute path names. For example, the specification of the c: drive in
    • build -rc c:\ode\.sandboxrc
    which would not be allowed in AIX, is legal in OS/2.
  • The path separator is the semicolon (;) character. For example, note the use of the semicolon in the following setenv statement
    • setenv OBJECTDIR ${sandbox}/obj/${CONTEXT};${OBJECTDIR}
  • ODE generates paths with UNIX-style forward slashes (/). For example,
    • ${.ALLSRC}
    is given a value like ./../src/file.c. If your compiler or other tool requires DOS-style backward slashes, you can use a variable modifier in the Makefile. For example, 
    ${.ALLSRC:S;/;\\;g}
    converts the string ../../src/file.c to ..\..\src\file.c.
  • The mksb and mklinks commands cannot populate the user's sandbox with symbolic links to backing build files.  Instead of creating symbolic links, mksb and mklinks copy files by default.
  • The workon command for OS/2 starts the command processor defined in the SHELL environment variable. If SHELL is not defined, it starts the command processor defined in the OS2_SHELL environment variable. If OS2_SHELL is not defined, it defaults to CMD.EXE.
  • No packaging tools are supported for OS/2.
  • Drive reference using the active current directory is not supported except for command line arguments where it makes sense.  For example, if on drive C: the current directory is temp which contains the file .sandboxrc, then from drive D: you attempt the command: mksb -rc C:.sandboxrc, this will work.  However, if drive references are made in config files, Makefiles, etc., the paths must be absolute rather than relative to the drive reference.
  • Ctrl-C and Ctrl-Break do not cause mk to run the .INTERRUPT target when a child process is running.

Use of #include "" in C/C++ Source Files

In most organizations of source files the C/C++ implementation file includes a local header file.  For example, we have main.c which includes file.h.  So the source for main.c is:
#include "file.h"
void main() { foo(); }
And the source for file.h is:
void foo();
In this example all three files reside in the same directory.  The problem comes in with ODE sandbox.  If a developer has only the file.h file in their sandbox and they try to compile main.c, the C/C++ compiler will include the file.h from the backing build.  By definition of the #include "" directive, the C/C++ compiler will begin searching for included files from the location of the current file being processed.  In this case, since main.c is in the backing build, then the C/C++ compiler finds the file.h in the backing build.

There are a couple of options to work around this problem.  The #include "file.h" can be changed to #include <file.h>.  Also it would be possible to export header files to a subdirectory in the export directory, then the #include "file.h" can be changed to #include "myincs/file.h".
 

Spaces in pathnames

With the exception of Windows NT, it is not customary to have blanks in pathnames, especially in development environments.  As a result, development tools (compilers, etc.) are not expected to support blanks in filenames, except on Windows NT.  ODE does not fully support the use of spaces in pathnames, so it is best to put sandboxes, backing builds, compiler tools, etc. in directories that have names that do not contain spaces.  For example, creating a backing build or sandbox on Windows NT under "C:\My Programs" can cause problems when using ODE tools.  Although some ODE tools can work in certain cases by enclosing appropriate paths in double quotes, there is no guarantee that they will work in all cases.

A generic workaround

One way to work around this problem is to create symbolic links (on Unix systems) or use the subst command (on Windows) to bypass the problematic directory.  For example:

ln -s "/tmp/directory with spaces/subdir" /tmp/shortcut
subst M: "C:\Program Files\subdir"

Then use the link or substituted drive when giving paths to ODE.  However, there are some restrictions on using symbolic links in ODE.  Since paths to backing builds and sandboxes should not use symbolic links, this restricts the above workaround somewhat.

Another solution on Windows is to use the DOS 8.3 short name of the directory.  For example, use PROGRA~1 instead of Program Files.  The short names can be viewed with dir /x.

Special NT support

Windows NT provides more support than other operating systems for blanks embedded in filenames.  However using double quotes often is still required to ensure that blanks are not interpreted as argument separators.  For NT only, several ODE tools have been modified to support blanks in filenames.  The intent is to support toolkits that have been installed on paths which have blanks embedded in filenames.  The support is still limited, but includes the following features and restrictions:
  • Windows NT only.  (Blanks are possible in UNIX filenames, but no UNIX development environment appears to support them.)
  • The ODE build environment itself (sandbox or backing build) must NOT have any blanks in filenames.
  • The toolkit can have blanks in the path to the toolkit, for example d:\Program Files\DevStudio
  • Leading blanks and trailing blanks in filenames are NOT supported.
  • Blanks in suffixes are NOT supported.
  • If ODE command flags include filenames, and the filenames have blanks, such flags must be double quoted.
  • Use only double quotes, as single quotes are not quaranteed to work
Several ODE tools are affected.  Click on the link for each tool for more details.
  • gendep: output *.u files will have double quotes around filenames if needed.
  • genpath: appropriate filenames in output will be double quoted if needed.
  • mkdep: *.u files used as input may have filenames with embedded blanks in dependency rules. Double quotes are not required.  The output depend.mk file will have quotes around filenames with embedded blanks.
  • mk: Dependency rules in makefiles are treated slightly differently.  Filenames with embedded blanks must be in double quotes so that the blanks are not treated as filename separators.

        • If a sandbox directory structure has more than one project or set in it, each project or set will have to be moved to its own sandbox structure. There is either a sets file or a projects file in the rc_files directory. It should be examined to understand how many sets or projects are in the sandbox.
        • Generally if there is a default project, its line in the projects file will resemble:
          • . projectname
          If sets are used instead, the equivalent in the sets file would be

          default projectname
          set projectname .

          The default project or set has its files directly under the src directory. These files can be left where they are. We are assuming that the sandbox will contain only this project or set after we are finished. The file sb.conf will be in the rc_files/projectname directory, and it should be moved to the rc_files directory. Then the projectname directory in rc_files may be removed.  The Buildconf, Buildconf.exp, and Buildconf.local files will be in the src/projectname directory and should be moved to the src directory. Then the projectname directory can be removed from the src directory.

        • If there are other projects, their lines in the projects file will have the form:
          • ./projectdir projectname
          The equivalent lines in the sets file would have the form:

          set projectname ./projectdir

          The project or set has its files under the src/projectdir directory.  These files should be moved to the src directory of a new sandbox. The sb.conf file for the projectname project or set will be in the rc_files/projectname directory, and it should be moved to the rc_files directory of the new sandbox.  Then the projectname directory in rc_files may be removed. The Buildconf, Buildconf.exp, and Buildconf.local files will be in the src/projectname directory and should be moved to src directory of the new sandbox.  Then the projectname directory can be removed from the src directory.

      • Files used to support the sets and projects features will be ignored by ODEi2.4, and may be removed.  These files are the sets or projects files in the rc_files directory.
      • The sb.conf file may need editing. You may remove the line that has the variable ode_sc. This line is ignored by ODEi2.4.
