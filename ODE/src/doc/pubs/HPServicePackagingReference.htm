About this Document

This document is intended to give the reader the necessary information to do Service Packaging on HP-UX version 11.0 or later.

See the ODE Documentation Index for a complete list of ODE documents and their purposes. It is recommended that the ODE Packaging Concepts and ODE Packaging Reference documents be read before reading the HP-UX Service Packaging Reference.

Introduction

ODE supports Initial Product Packaging(IPP) and Service Packaging(SP) on HP-UX. The tool used to create packages on HP-UX is swpackage. Though IPP is supported on HP-UX versions 10 or lower, SP is supported only on versions 11.0 or later. IPP creates a full and complete package, containing the entire contents of a product. Service packaging creates a patch which generally contains a partial, or incremental set of files used to update or refresh an existing software installation. Swpackage uses the PSF (Product Specification File) to organize files into products, subproducts and filesets.

ODE Targets for Service Packaging

ODE requires the same set of targets for Service Packaging as for Initial Product Packaging, namely gather_all, parse_all and runpkgtool_all. The same file structure that is currently being used for IPP can be used to create a patch in Service Packaging and the CMF need not be changed except while adding new filesets or files as a part of the patch. The major difference is that SP will use an extra set of makefile variables which will be discussed below.

A typical patch creation process involves creating a sandbox, populating the sandbox with the files which need to be changed for the patch and running the following targets after the files are changed.

'mk': Run as mk
This pass builds the modified source files.

'gather_all': Run as mk gather_all PKG_CLASS=sp
It runs the instdep_all target. This target populates the shiptree with the changed/modified files. Note: The shiptree can be populated with the modified files either by issuing mk gather_all PKG_CLASS=sp or mk instdep_all command.

'parse_all': Run as mk parse_all PATCH_NAME=xxxx
This target parses the CMF and generates the PSF which will be used by the packaging tool swpackage. Since the PKG_CLASS is set to "sp", the resultant PSF will contain the entries for only the files present in the shiptree of the sandbox. This pass uses the makefile variables PATCH_NAME, PATCH_NEW_FILESETS, PATCH_ANCESTOR_LIST, PATCH_SUPERSEDES_LIST, PATCH_NEW_LINKS and PATCH_NEW_DIRS which are discussed in the next section.

'runpkgtool_all': Run as mk runpkgtool_all PKG_CLASS=sp
This target runs the actual packaging command swpackage. This command is invoked through the script Spti_swpackage and an installable patch is created based on the contents of the PSF.

Makefile Variables

The following makefile variables are used in Service Packaging. Some of these variables are required and some are optional. These makefile variables could be either defined in a makefile or on the command line itself.
 
Makefile Variable Definition Required/Optional Format
PATCH_NAME This makefile variable generates the tag for the patch in the PSF. This is a required makefile variable. The patch tag should be different from the original product tag as recommended by HP. Required string
PATCH_NEW_FILESETS This makefile variable contains a list of colon separated filesets. All the filesets which don't exist in the base product and are being newly introduced in the patch should be listed here. For more information on this variable, check Support for New Filesets. Optional colon separated list of fileset names. For Example: odehellobin:odehellodoc
PATCH_ANCESTOR_LIST This makefile variable contains a list of filesets with each entry being an ancestor to a particular fileset listed in the PSF. It tells the SD that it is replacing that particular fileset. For more information on this variable check Support for Ancestor and Supersedes. Optional Each entry in this variable corresponds to a fileset in the PSF. All the entries should be separated by : (colon)
Each entry will have a fileset and a corresponding list of superseding filesets. The fileset and the list should be separated by @. In each entry, the superseding filesets should be separated by ; (semicolon). Each superseding fileset will be in the format Product.Filset,Version 
PATCH_SUPERSEDES_LIST This makefile variable contains a list of filesets with each entry being a supersede to a particular fileset listed in the PSF. For more information on this variable check Support for Ancestor and Supersedes. Optional Same as PATCH_ANCESTOR_LIST
PATCH_NEW_LINKS This makefile variable contains a list of colon separated symbolic links to be included in the patch. For more information on this variable and the support for symbolic links, check Support for Symbolic Links. Optional colon separated list of links. Each link is of format sourceDir/sourceFile where sourceDir and sourceFile are attributes of the file stanza that represents a symbolic link in the CMF.
PATCH_NEW_DIRS This makefile variable contains a list of colon separated directories to be included in the patch. For more information on this variable and the support for directories, check Support for Directories. Optional colon separated list of directories. Each directory should exactly match sourceDir attribute of the file stanza that represents a directory in the CMF.
PKG_CONTROL_DIR This variable represents the location of the control files. Required valid path to a directory
PKG_OUTPUT_DIR This variable represents the directory where the patch is created. The value of this variable is used to specify -d option of swpackage. Required valid path to a directory
PKG_TOOL_FLAGS This variable is used to specify all the supported flags that can be passed to swpackage Optional \"option1 option2 ...\"".
E.g.: "\"-d /test/dir -f file\""

Support for New Filesets

ODE supports the creation of new filesets in a patch. All the new filesets (created in the patch but not present in the base product) should be listed as a colon separated list in the makefile variable PATCH_NEW_FILESETS. The tags is_patch true, ancestor and supersedes will not be generated for these filesets.

Support for Ancestor and Supersedes

ODE will generate ancestor and supersedes tags in the PSF.

ODE will generate the ancestor tag in all the filesets in the PSF based on the values specified in the makefile variable PATCH_ANCESTOR_LIST. This variable should contain a colon separated list with each entry containing a fileset and its corresponding ancestors. If this variable is not set or if it does not have an entry for a fileset, ODE will generate this tag dynamically for the fileset (if the fileset is not new) in the format product.fileset where product refers to the name of the base product for which the patch is applied and fileset refers to the name of the fileset for which this tag is generated. In IPP packaging, the ancestor tag is generated from the value of the "Sup" requisite as specified in the requisites attribute in the RequisitesInfo stanza of the CMF. But in SP, this value will be ignored.

E.g.: It can be specified in the makefile as
PATCH_ANCESTOR_LIST=odehellobin@odehello.binfileset:odehellodoc@odehello.docfileset;odetestdoc.testfileset

ODE will generate the supersedes tag in all the required filesets in the PSF based on the values specified in the makefile variable PATCH_SUPERSEDES_LIST. This variable should contain a colon separated list with each entry containing a fileset and its corresponding supersedes. Note that this tag will be generated for only those filesets listed in this variable and hence no tag will be generated for a fileset if this variable is not set or if it does not have an entry for this fileset or if the fileset is new.

E.g.: It can be specified in the makefile as
PATCH_SUPERSEDES_LIST=odehellobin@odetest.fileset1,fr=1.0;odedev.fileset2,fr>1.1:odehelodoc@odebld.fileset3,fr>2.0;odeship.fileset4,fr<1.2

Important: Note that the ancestor and supersedes tags will not be generated if the fileset is listed in PATCH_NEW_FILESETS, which indicates that the fileset is new even if it is listed in any of the above two variables.

Support for Symbolic Links

ODE supports the packaging of symbolic links in a patch. Swpackage packages this link irrespective of it pointing to a valid file or not.

The symbolic links will be supported through a makefile variable PATCH_NEW_LINKS. This variable will contain a colon separated list of links to be packaged. A link listed in the CMF but missing in this variable will be ignored. All the links listed in this variable will be created in the shiptree and get included in the patch by swpackage.

E.g.: It can be specified in the makefile as
PATCH_NEW_LINKS=/opt/odehello/bin/link1:/opt/odehello/html/link2

Note that each entry in this variable should exactly match the combination of sourceDir and sourceFile attributes for a particular link in the CMF. Also, if the fileset which contains the link is new (listed in PATCH_NEW_FILESETS) the above checking will not be done. In this case, the link will be entered into the PSF and get created in the shiptree.

Support for Directories

ODE will ignore all the directory entries in the CMF while generating the PSF. PATCH_NEW_DIRS is a makefile variable containing a list of colon separated directories. Only the entries for those directories listed in this variable will be generated in the PSF.

E.g.: It can be specified in the makefile as
PATCH_NEW_DIRS=/opt/odehello/bin/:/opt/odehello/html/

Note that each entry in this variable should exactly match the sourceDir attribute for a particular directory in the CMF.
Also, if a new fileset (specified in PATCH_NEW_FILESETS) contains a directory, then that directory will be generated in the PSF irrespective of its existence in PATCH_NEW_DIRS.

Hard Coded Entries in the PSF

The tag layout_version will be set to 1.0 at the depot level in the PSF (See note below).
The tag is_patch will be set to true once at the product level and also in all the filesets in the PSF provided that the fileset is not new (not listed in PATCH_NEW_FILESETS).

Note: Beginning with HP-UX 11.0, patching was significantly enhanced by extending the abilities of the Software Distributor tools. The price of these new abilities was to introduce new attributes to the SD objects. The layout_version attribute is used to differentiate between the old and new, with 10.X depots identified by the value 0.8 and 11.X depots marked by 1.0.

Limitations

  • Deletion of files in a product by applying a patch will not be supported in this version.
  • Service Packaging is supported only on HP-UX versions 11 or higher.
  • ODE will not check the validity of ancestors and supersedes for any fileset.
